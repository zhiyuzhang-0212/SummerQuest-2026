# A1 评估补充说明（批改助教）

> 本文件是对 A1 **评估方式**的补充,不改变 [`README.md`](README.md) 中的作业要求。
> 日志的格式说明见 [`README.md`](README.md) 的《实验日志格式》;本文只说明评分标准与评测方式。

## 评估怎么做

- **代码部分**用 `uv run pytest` 客观判定(参考 `README.md` 的“代码接口”一节),通过的测试
  越多得分越高。
- **书面题与实验部分**借助大模型对提交的报告和日志做**整体、灵活**的评估,再由批改助教
  复核。不逐字卡格式,重点看是否真正完成、结论是否可信、分析是否到位。

## 评估主要看什么(参考权重,非逐项硬扣)

| 方面 | 大致权重 | 看什么 |
|---|---|---|
| 代码实现(pytest) | ~40% | 21 个 adapter 对应测试的通过情况 |
| 书面题 | ~10% | `unicode`、AdamW 显存/FLOPs/时间核算等的正确性与推导 |
| 训练与实验 | ~35% | TinyStories 是否训练跑通、val loss 水平;学习率扫、batch size、四个消融、OWT、文本生成是否完成并有分析 |
| 报告与记录质量 | ~15% | 实验记录、loss 曲线、结论是否清晰、可信、可复现 |

权重仅供同学了解重点,实际评估以整体质量为准。

## 提交格式与评测的灵活性

`logs/` 与 `summary.json` 的**格式说明见 [`README.md`](README.md) 的《实验日志格式》**（核心
字段为要求，文件名与组织为推荐）。评测过程借助大模型做整体判断、相对灵活:重点看内容是否
真实完成、结论是否可信,不会因字段或文件名的细小差异直接扣分。

报告里另外建议给出(便于评估):

- **loss 曲线**(能同时体现步数和墙钟时间更好),放 `assets/` 并在报告里引用;
- 学习率扫注明哪个 run 发散;四个消融各给一段对比与分析;
- 文本生成贴一段样本并简评流畅度。

## 诚信提醒

因为单文件不超过 5 MiB,提交里一般不含 checkpoint 和数据,日志是主要证据。请**如实记录**
实验结果;批改可能对个别同学做抽查,请其用自己的脚本现场重跑若干步以核对曲线。
